# OpenRisk 현재 로직 및 데이터 정리

> 작성일: 2026-04-26
> 기준: 현재 코드 구현(`app/api/v2/analyze/route.ts`, `lib/v2/*`, `lib/categories.ts`, `scripts/*`)
> 목적: 지금 서비스가 실제로 사용하는 분석 로직, 데이터 테이블, 외부 API, 보조 기능을 한 문서에서 확인하기 위한 정리

---

## 1. 한 줄 요약

OpenRisk는 사용자가 선택한 좌표와 업종을 기준으로 H3 해상도 9 그리드의 반경 500m 데이터를 모아, 경쟁·유동·임대료·생존·앵커 지표를 업종별 가중치로 합산해 0~100 리스크 점수를 계산한다. 점수가 높을수록 창업 위험이 크다는 뜻이다.

---

## 2. 현재 서비스 흐름

```text
사용자 위치 검색
  -> GET /api/search
  -> Kakao 주소/키워드 검색
  -> 서울·경기·인천·부산 결과만 노출

업종 선택 후 분석
  -> POST /api/v2/analyze
  -> 좌표 검증, 지원 지역 판별
  -> H3 셀 목록 생성
  -> Supabase 그리드 데이터 조회
  -> 500m 실제 거리 필터링
  -> 5대 지표 계산
  -> 상권 유형, 시간대 패턴, 생존 신호 계산
  -> 리스크 점수, 해석, 리스크 카드 생성

AI 경고 요약
  -> POST /api/ai/summary
  -> v2 분석 결과를 축약
  -> OpenAI 모델로 경고형 JSON 리포트 생성
```

주요 화면/경로:

| 경로 | 역할 |
|------|------|
| `app/home-b/page.tsx` | 메인 검색 및 최근 게시글 |
| `app/result-b/page.tsx` | 현재 분석 결과 화면 |
| `app/api/search/route.ts` | 위치 자동완성 |
| `app/api/v2/analyze/route.ts` | 핵심 분석 API |
| `app/api/ai/summary/route.ts` | AI 경고 요약 |
| `app/board/*` | 커뮤니티 |
| `app/api/stats/*` | 방문자 통계 |

## 3. 지원 지역과 분석 단위

### 3.1 지원 지역

현재 API는 좌표 범위로 1차 지원 지역을 판별한다.

| 지역 | 위도 범위 | 경도 범위 | 비고 |
|------|-----------|-----------|------|
| 서울 | 37.41~37.72 | 126.75~127.27 | 우선 지원 |
| 경기 | 36.85~38.35 | 126.20~127.95 | 데이터 커버리지 의존 |
| 인천 | 37.30~37.65 | 126.35~126.95 | 데이터 커버리지 의존 |
| 부산 | 34.95~35.35 | 128.75~129.35 | 코드상 지원, DB 데이터 없으면 준비중 응답 |

지원 지역 밖이면 `422`과 함께 `현재는 서울·경기·인천·부산 지역만 지원합니다.`를 반환한다.

### 3.2 분석 단위

| 항목 | 값 |
|------|----|
| H3 해상도 | 9 |
| 분석 반경 | 500m 고정 |
| H3 후보 셀 | 중심 셀 기준 `gridDisk` |
| 최종 사용 셀 | 각 셀 중심점이 실제 500m 이내인 데이터만 사용 |
| 좌표 거리 계산 | Haversine |

관련 파일:

| 파일 | 역할 |
|------|------|
| `lib/h3.ts` | H3 변환, 반경 셀 생성, 거리 계산 |
| `app/api/v2/analyze/route.ts` | 반경 필터링 및 지표 집계 |

## 4. 핵심 데이터 테이블

### 4.1 `grid_store_counts`

H3 셀별 점포 집계 테이블이다. 현재 경쟁 지표, 상권 유형, 생존 신호의 핵심 입력이다.

| 컬럼 | 용도 |
|------|------|
| `h3_id` | H3 셀 ID, 기본 키 |
| `center_lat`, `center_lng` | 셀 중심 좌표 |
| `region` | 서울, 경기, 인천, 부산 등 |
| `district` | 시군구 |
| `store_counts` | 업종별 점포 수 JSON |
| `total_count` | 전체 점포 수 |
| `prev_period_count` | 이전 기간 점포 수 |
| `closure_count` | 폐업 점포 수 |
| `opening_count` | 개업 점포 수 |
| `period` | 데이터 기준 기간 |

주요 사용:

| 사용처 | 내용 |
|--------|------|
| 경쟁 지표 | 반경 500m 내 `store_counts[targetCategory]` 합산 |
| 전체 점포 | 반경 500m 내 `total_count` 합산 |
| 상권 유형 | 음식점/카페/주점 비율, 생활서비스 비율 계산 |
| 생존 신호 | `closure_count`, `opening_count`, `prev_period_count`로 폐업률/개업률 계산 |
| 유동 보정 | 교통 지수 계산 스크립트에서 점포 밀도 보정값으로 사용 |

데이터 적재 관련:

| 파일/API | 역할 |
|----------|------|
| `scripts/import-store-csv.ts` | 서울·경기·인천 소상공인 CSV를 H3로 집계 |
| `scripts/import-busan-stores.ts` | 부산 소상공인 CSV를 H3로 집계 |
| `app/api/admin/import-stores/route.ts` | CSV 기반 점포 카운트 보정 API |
| `app/api/admin/import-closure/route.ts` | `closure_update.json`으로 개폐업 수 업데이트 |

### 4.2 `grid_traffic`

H3 셀별 유동인구 지수 테이블이다. 운영 API는 여러 필드명을 호환 처리한다.

| 컬럼 | 용도 |
|------|------|
| `h3_id` | H3 셀 ID |
| `traffic_estimated` 또는 `traffic_index` | 최종 유동 지수 |
| `traffic_morning` 또는 `time_morning` | 오전 비중 |
| `traffic_day` 또는 `time_day` | 낮 비중 |
| `traffic_night` 또는 `time_night` | 저녁/야간 비중 |
| `weekend_ratio` | 주말 비중 |
| `nearest_subway_id`, `nearest_subway_name`, `subway_distance` | 근접 지하철 정보 |
| `nearest_bus_stops` | 근접 버스정류장 정보 |
| `subway_component`, `bus_component`, `poi_factor` | 교통 지수 구성 요소 |
| `period` | 데이터 기준 기간 |

주요 사용:

| 사용처 | 내용 |
|--------|------|
| 유동 지표 | 반경 500m 내 유효 셀의 평균 유동 지수 |
| 시간대 패턴 | 오전/낮/저녁 비중 정규화 |
| 피크타임 | 세 시간대 중 최대값 |
| 주말 비중 | 평균값 사용, 0~100 저장값이면 0~1로 변환 |

관련 스크립트:

| 파일 | 역할 |
|------|------|
| `scripts/calculate-traffic-index.ts` | 지하철+버스+점포밀도로 `grid_traffic` 생성 |
| `scripts/import-seoul-population.ts` | 서울 생활인구 H3 보정 |
| `scripts/import-gyeonggi-population.ts` | 경기 생활인구 H3 보정 |
| `scripts/import-incheon-taxi.ts` | 인천 택시/유동 보정 |
| `scripts/import-sdot-traffic.ts` | S-DoT 유동인구 보정 |
| `scripts/import-busan-bus.ts` | 부산 버스 데이터 |
| `scripts/import-busan-subway.ts` | 부산 지하철 데이터 |

### 4.3 `district_rent`

지역별 임대료 테이블이다. 분석 API는 주소 역지오코딩 결과의 구/군명으로 정확 매칭을 먼저 시도하고, 실패하면 부분일치 후보를 점수화한다.

| 컬럼 | 용도 |
|------|------|
| `legal_dong_code` | 고유 코드 |
| `district_name` | 구/군 또는 상권명 |
| `region` | 서울, 경기, 인천, 부산 |
| `avg_rent_per_pyeong` | 평균 임대료, 만원/평 기준 |
| `rent_level` | `low`, `medium`, `high` |
| `sample_count` | 표본 수 |
| `period` | 기준 기간 |

임대료 계산 우선순위:

| 순서 | 방식 |
|------|------|
| 1 | `district_name` 정확 매칭 |
| 2 | `district_name` 부분일치 후보 점수화 |
| 3 | 같은 지역 임대료 중앙값 |
| 4 | 기본값 20만원/평 |

관련 스크립트:

| 파일 | 역할 |
|------|------|
| `scripts/import-rent-data.ts` | 서울·경기·인천 2025년 3분기 상업용 부동산 임대료 적재 |
| `scripts/import-busan-rent.ts` | 부산 임대료 적재 |

### 4.4 `subway_stations`, `bus_stops`

교통 지수와 앵커 시설 판단에 사용한다.

| 테이블 | 주요 컬럼 | 사용처 |
|--------|-----------|--------|
| `subway_stations` | 역명, 노선, 좌표, 승하차, 시간대 데이터 | 유동 지수, 앵커 지하철 |
| `bus_stops` | 정류장명, 좌표, 승하차, 노선 수 | 유동 지수 |

RPC:

| 함수 | 역할 |
|------|------|
| `find_nearest_subway(p_lat, p_lng, p_limit)` | 가장 가까운 지하철역과 거리 반환 |

현재 분석 API는 가장 가까운 지하철역이 1.2km 이내일 때만 앵커로 인정한다.

### 4.5 외부 API 데이터

| API | 사용처 |
|-----|--------|
| Kakao Local 좌표->주소 | 분석 좌표의 주소, 지역, 구/군 추출 |
| Kakao 주소/키워드 검색 | 검색 자동완성 |
| Kakao 키워드 검색 `스타벅스` | 800m 내 스타벅스 개수와 최근접 거리 |
| Kakao 카테고리 `MT1` | 1.2km 내 대형마트 검색 |
| OpenAI API | AI 경고 요약 생성 |

백화점은 카카오 검색 대신 코드에 주요 백화점 좌표 목록을 두고 1.2km 이내 최근접 시설을 계산한다.

## 6. 분석 API 세부 로직

### 6.1 요청과 응답

요청:

```json
{
  "lat": 37.5,
  "lng": 127.0,
  "targetCategory": "cafe"
}
```

필수 검증:

| 검증 | 실패 응답 |
|------|-----------|
| `lat`, `lng`, `targetCategory` 존재 | 400 |
| 좌표가 숫자이고 유한값 | 400 |
| 위도/경도 범위 | 400 |
| 지원 지역 좌표 범위 | 422 |
| 업종 키 유효성 | 400 |
| 반경 내 점포 또는 유동 데이터 없음 | 422, 지역 데이터 준비중 메시지 |

응답 핵심 구조:

| 필드 | 내용 |
|------|------|
| `location` | 좌표, 주소, 지역, 구/군 |
| `analysis` | 리스크 점수, 레벨, 상권 유형, 업종명, 점수 분해 |
| `metrics.competition` | 경쟁 지표 |
| `metrics.traffic` | 유동 지표 |
| `metrics.cost` | 임대료 지표 |
| `metrics.survival` | 생존 지표 |
| `anchors` | 지하철, 스타벅스, 마트, 백화점 |
| `interpretation` | 요약, 리스크/기회 문장, 쉬운 해석, 점수 기여도 |
| `riskCards` | 우선순위 상위 3개 리스크 카드 |
| `dataQuality` | 점포/유동 기준일, 커버리지 |
| `h3Cells`, `centerH3` | 지도 표시용 H3 셀 |
| `integrity` | 설정된 경우 서버 서명 |

### 6.2 캐시와 레이트리밋

| API | 제한 | 캐시 |
|-----|------|------|
| `/api/search` | IP별 45회/분 | 5분, 최대 2,000개 |
| `/api/v2/analyze` | IP별 20회/분 | 5분, 최대 1,000개 |
| `/api/ai/summary` | IP별 8회/분 | 별도 캐시 없음 |

레이트리밋은 Upstash Redis 환경변수가 있으면 Redis를 사용하고, 없으면 서버 메모리 `Map`을 사용한다.

## 7. 지표 계산 로직

### 7.1 경쟁 지표

입력:

| 값 | 산출 방식 |
|----|-----------|
| `total` | 반경 내 `total_count` 합계 |
| `sameCategory` | 반경 내 `store_counts[targetCategory]` 합계 |
| `density` | `min(1, sameCategory / 20)` |
| `hasCategoryData` | 해당 카테고리 키가 DB에 존재하는지 |

동적 임계값:

| 값 | 의미 |
|----|------|
| `p50` | 같은 지역·업종 H3 셀의 중앙값 |
| `p75` | 75백분위 |
| `p90` | 90백분위 |
| `sampleSize` | 표본 수 |

동적 프로필은 `grid_store_counts` 전체에서 지역·업종별로 계산하고 12시간 캐시한다. 표본이 10개 미만이면 기본값을 사용한다.

경쟁 레벨:

| 조건 | 레벨 |
|------|------|
| `sameCategory <= p50 또는 기본 5` | `low` |
| `sameCategory <= p75 또는 기본 15` | `medium` |
| 그 외 | `high` |

정규화 점수:

| 조건 | 점수 |
|------|------|
| `sameCategory <= lowThreshold` | 0~30 선형 |
| `sameCategory >= highThreshold` | 100 |
| 사이 구간 | 30~100 선형 |

### 7.2 유동 지표

운영 API는 반경 내 `grid_traffic` 유효 셀의 평균 지수를 사용한다.

| 항목 | 계산 |
|------|------|
| `index` | 유효 셀의 `traffic_estimated` 또는 `traffic_index` 평균 |
| `timePattern` | 오전/낮/저녁 합계를 100%로 정규화 |
| `weekendRatio` | 유효 셀 평균, 1보다 크면 퍼센트로 보고 100으로 나눔 |
| `peakTime` | 오전/낮/저녁 중 최대값 |

레벨:

| 유동 지수 | 레벨 | 표시 |
|-----------|------|------|
| `<= 1` | `very_low` | 매우 낮음 |
| `<= 6` | `low` | 낮음 |
| `<= 13` | `medium` | 보통 |
| `<= 40` | `high` | 높음 |
| `> 40` | `very_high` | 매우 높음 |

리스크 정규화는 역방향이다.

| 조건 | 점수 |
|------|------|
| `index <= 3` | 100 |
| `index >= 20` | 25 |
| 사이 구간 | 100에서 25로 선형 감소 |

유동 데이터가 기본 패턴 `{ morning: 33, day: 34, night: 33 }`이면, 상권 유형과 역세권 여부로 시간대 패턴을 추정한다.

| 상권 유형 | 조건 | 패턴 | 피크 | 주말 비중 |
|-----------|------|------|------|-----------|
| `A_주거` | 기본 | 30/25/45 | night | 0.45 |
| `B_혼합` | 역 300m 내 | 35/30/35 | morning | 0.35 |
| `B_혼합` | 일반 | 30/35/35 | day | 0.40 |
| `C_상업` | 역 300m 내 | 30/40/30 | day | 0.30 |
| `C_상업` | 일반 | 25/45/30 | day | 0.35 |
| `D_특수` | 기본 | 20/30/50 | night | 0.55 |

### 7.3 임대료 지표

| 항목 | 계산 |
|------|------|
| `avgRent` | 구/군 매칭값, 지역 중앙값, 또는 기본 20만원/평 |
| `districtAvg` | 구/군 매칭 시 해당 평균 |
| `source` | `district`, `regional_median`, `default` |
| `thresholds` | 지역 임대료 p50, p75, p90 |
| `percentile` | 지역 내 백분위 순위 |

임대료 레벨:

| 프로필 있음 | 레벨 |
|-------------|------|
| `avgRent >= p75` | `high` |
| `avgRent >= p50` | `medium` |
| 그 외 | `low` |

프로필이 없으면 저장된 `rent_level`을 우선 사용하고, 없으면 `<=15 low`, `<=40 medium`, 그 외 `high`를 사용한다.

정규화 점수:

| 조건 | 점수 |
|------|------|
| `percentile` 존재 | 백분위 값을 그대로 0~100 점수로 사용 |
| `avgRent <= p50 또는 기본 15` | 0~30 선형 |
| `avgRent >= p90 또는 기본 40` | 100 |
| 사이 구간 | 30~100 선형 |

### 7.4 앵커 지표

앵커 시설은 없을수록 위험 점수가 높다.

| 앵커 | 인정 기준 |
|------|-----------|
| 지하철 | 최근접 역 1.2km 이내 |
| 스타벅스 | Kakao 키워드 검색 800m 이내 |
| 대형마트 | Kakao `MT1` 1.2km 이내, 익스프레스/슈퍼/프레시 제외 |
| 백화점 | 코드 내 주요 백화점 좌표 기준 1.2km 이내 |

정규화 점수:

| 조건 | 점수 |
|------|------|
| 앵커 없음 | 80 |
| 지하철 없음 | +40 |
| 지하철 100m 이내 | +0 |
| 지하철 300m 이내 | +10 |
| 지하철 500m 이내 | +20 |
| 지하철 500m 초과 | +40 |
| 스타벅스 없음 | +15 |
| 스타벅스 1~2개 | +5 |
| 스타벅스 3개 이상 | +0 |
| 대형마트 없음 | +10 |
| 백화점 없음 | +5 |
| 최종 상한 | 80 |

### 7.5 상권 유형

상권 유형은 점포 구성, 유동 레벨, 주말 비중, 지하철 접근성으로 결정한다.

상업형 점포:

```text
한식, 양식, 일식, 중식, 치킨, 피자, 패스트푸드,
카페, 베이커리, 디저트, 술집/바
```

주거형 점포:

```text
편의점, 세탁소, 약국, 슈퍼마켓, 미용실
```

판정 순서:

| 순서 | 조건 | 결과 |
|------|------|------|
| 1 | `weekendRatio > 0.5` | `D_특수` |
| 2 | 지하철 300m 이내 + 상업형 비율 70% 초과 | `D_특수` |
| 3 | 유동 `high` 이상 + 상업형 비율 60% 초과 | `C_상업` |
| 4 | 전체 점포 20개 미만 또는 주거형 비율 40% 초과 | `A_주거` |
| 5 | 그 외 | `B_혼합` |

상권 유형 패널티:

| 유형 | 패널티 |
|------|--------|
| `A_주거` | 0 |
| `B_혼합` | +3 |
| `C_상업` | +10 |
| `D_특수` | +15 |

### 7.6 생존 지표

실제 개폐업 데이터가 있으면 실제값을 사용하고, 없으면 추정 로직으로 대체한다.

실제 데이터 사용 조건:

```text
sum(prev_period_count || total_count) > 0
```

주의: 현재 코드는 `prev_period_count`가 없으면 `total_count`를 이전 점포 수처럼 대체한다. 따라서 개폐업 컬럼이 0 또는 null인 상태에서도 점포 데이터가 있으면 추정 로직으로 넘어가지 않고 실제 데이터 분기로 계산될 수 있다.

실제 계산:

| 항목 | 공식 |
|------|------|
| 폐업률 | `closure_count 합계 / prev_count 합계 * 100` |
| 개업률 | `opening_count 합계 / prev_count 합계 * 100` |
| 순증감 | `opening_count 합계 - closure_count 합계` |

폐업 리스크:

| 폐업률 | risk |
|--------|------|
| `< 30%` | `low` |
| `< 50%` | `medium` |
| `>= 50%` | `high` |

추정 로직 입력:

| 입력 | 의미 |
|------|------|
| 업종 기본 생존율 | `lib/v2/closure-risk.ts`의 업종별 1년 생존율 |
| 경쟁 밀도 | 동종 점포 / 전체 점포 |
| 유동 레벨 | `very_low`~`very_high` |
| 임대료 레벨 | `low`/`medium`/`high` |
| 상권 유형 | A/B/C/D |

추정 보정:

| 요소 | 보정 |
|------|------|
| 경쟁 밀도 0.5 초과 | 최대 +20% |
| 경쟁 밀도 0.2 미만 | -5% |
| 유동 `very_high` | -5% |
| 유동 `high` | -3% |
| 유동 `low` | +3% |
| 유동 `very_low` | +5% |
| 임대료 `low` | -5% |
| 임대료 `high` | +10% |
| `A_주거` | x0.9 |
| `B_혼합` | x1.0 |
| `C_상업` | x1.2 |
| `D_특수` | x1.3 |

추정 폐업률은 5~80% 범위로 제한한다.

### 7.7 시간대 보정

업종별 최적 피크타임과 실제 피크타임이 일치하면 리스크가 줄고, 불일치하면 증가한다.

| 업종군 | 최적 피크 |
|--------|-----------|
| 한식, 중식, 패스트푸드, 베이커리, 약국, 세탁소, 미용실, 네일, 슈퍼, 디저트 | day |
| 양식, 일식, 술집, 치킨, 피자, 헬스장, 학원 | night |
| 카페, 편의점 | morning |

보정:

| 조건 | 점수 |
|------|------|
| 최적 피크와 실제 피크 일치 | -5 |
| 주말 민감 업종인데 주말 비중 < 0.3 | +8 |
| 주말 민감 업종인데 주말 비중 > 0.45 | -3 |
| 평일 민감 업종인데 주말 비중 > 0.5 | +8 |
| 평일 민감 업종인데 주말 비중 < 0.3 | -3 |
| 그 외 피크 불일치 | +3 |

주말 민감 업종:

```text
술집/바, 양식, 디저트, 카페
```

평일 민감 업종:

```text
한식, 중식, 세탁소, 약국
```

## 8. 종합 리스크 점수

공식:

```text
기본 점수 =
  경쟁 위험 점수 * 업종별 경쟁 가중치
  + 임대료 위험 점수 * 업종별 임대료 가중치
  + 생존 위험 점수 * 업종별 생존 가중치
  + 유동 위험 점수 * 업종별 유동 가중치
  + 앵커 위험 점수 * 업종별 앵커 가중치

최종 점수 =
  기본 점수
  + 시간대 보정
  + 상권 유형 패널티

최종 점수는 0~100으로 clamp 후 반올림한다.
```

리스크 레벨:

| 점수 | 레벨 | 의미 |
|------|------|------|
| 0~29 | `LOW` | 낮음 |
| 30~49 | `MEDIUM` | 보통 |
| 50~69 | `HIGH` | 높음 |
| 70~100 | `VERY_HIGH` | 매우 높음 |

응답에는 `scoreBreakdown`으로 정규화 점수, 보정값, 가중 기본 점수, 최종 점수를 함께 내려준다.

|------|------|

주요 카드 조건:

| 카드 | 조건 |
|------|------|
| `3중 리스크` | 경쟁 과밀 + 임대료 높음 + 유동 낮음 |
| `파이 쪼개기 구조` | 경쟁 과밀 + 유동 낮음 |
| `이중 압박 구조` | 임대료 높음 + 경쟁 과밀 |
| `상권 축소 중` | 생존 리스크 높음 + 순감소 |
| `유입 동선 없음` | 유동 낮음 + 앵커 없음 |
| `역세권 아님` | 지하철 거리 500m 초과 |
| `시즌 의존 상권` | `D_특수` |

| `data/bus/` | 서울·경기·인천 버스 승하차 데이터 |
| `data/보정/` | S-DoT, 행정동 유동인구, 인천 유동 보정 데이터 |
| `data/판단/` | 철도 승하차, 역 위치 등 판단용 원천 파일 |
| `data/*_h3_data.json` | 지역별 H3 보정 결과 |

## 14. 운영상 주의할 점

1. 현재 분석 API 기준 문서는 이 파일을 우선한다. 기존 `docs/등급계산로직.md`, `docs/지표현황.md`, `docs/specs/*`에는 v1 또는 초기 v2 기준이 일부 남아 있다.
2. `scripts/v2-schema.sql`의 스키마와 운영 코드가 읽는 필드명이 일부 다르다. 운영 API는 `traffic_estimated`/`traffic_index`, `traffic_morning`/`time_morning`을 모두 처리한다.
3. 부산은 코드상 지원 지역에 포함되어 있지만, 분석 성공 여부는 Supabase의 `grid_store_counts`와 `grid_traffic` 데이터 적재 여부에 달려 있다.
4. 임대료는 구/군 또는 하위시장 평균이므로 같은 구 안에서도 실제 임대료 차이가 클 수 있다.
5. 유동인구는 교통/보정 데이터 기반 상대 지수이며, 실제 보행자 수나 매출 예측값이 아니다.
6. 폐업률은 실제 개폐업 카운트가 있으면 사용하고, 없으면 업종·상권·유동·임대료 기반 추정값이다.
7. AI 요약은 분석 결과를 다시 표현하는 기능이다. 점수 계산 자체에는 AI가 관여하지 않는다.
